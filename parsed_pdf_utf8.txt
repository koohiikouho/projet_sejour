3.  Developer  Documentation  
3.1.  Architecture  Overview:  
The  Sojourn  Compass  application  is  built  on  an  Offline-First,  
Service-Oriented
 
Architecture
 
(SOA).
 
Given
 
the
 
variable
 
nature
 
of
 
European
 
public
 
internet
 
access,
 
the
 
system
 
relies
 
on
 
a
 
local
 
SQLite
 
cache
 
as
 
its
 
primary
 
data
 
layer,
 
synchronized
 
asynchronously
 
with
 
a
 
Firebase
 
Cloud
 
Firestore
 
backend.
 
System  Architecture  Diagram:  
 
Login  Process:  Users  open  the  app  and  trigger  an  authentication  check.  
Unauthenticated
 
users
 
are
 
served
 
the
 
login
 
prompt,
 
verifying
 
credentials
 
against
 
the
 
backend.
 
Success
 
yields
 
a
 
token,
 
distinguishing
 
standard
 
pilgrims
 
from
 
coordinators.
 
Group  Tracking  Process:  Upon  opening  the  Group  Tracker,  background  
geolocation
 
services
 
collect
 
real-time
 
coordinates.
 
The
 
backend
 
aggregates
 
this
 
point
 
data
 
and
 
renders
 
it
 
onto
 
the
 
map.
 
Team
 
leaders
 
use
 
this
 
to
 
conduct
 
digital
 
headcounts
 
and
 
locate
 
wandering
 
members.
 
Daily  Itinerary  Process:  Users  select  the  itinerary  tab,  pulling  offline  
data
 
from
 
the
 
local
 
database.
 
If
 
participants
 
spot
 
outdated
 
information,
 
they
 
can
 
submit
 
an
 
incorrect
 
information
 
report
 
(feedback
 
loop)
 
which
 
coordinators
 
receive
 
instantly
 
to
 
issue
 
updates.
 
Announcement  Retrieval:  Dual-layer  approach  where  the  app  polls  the  
backend
 
for
 
the
 
latest
 
published
 
messages,
 
stores
 
them
 
in
 
the
 
local
 
database,
 
and
 
renders
 
them.
 
This
 
guarantees
 
access
 
during
 
cellular
 
blackouts
 
on
 
foreign
 
buses.
 
AR  Process:  Users  invoke  the  camera  tab,  point  the  device  at  a  historical  
artifact,
 
and
 
the
 
system
 
performs
 
tracking
 
and
 
recognition
 
to
 
overlay
 
translated
 
historical
 
contexts
 
directly
 
onto
 
the
 
screen.

The  proprietary  logic  is  written  in  Dart  using  the  Flutter  framework ,

●  /lib/models/: Contains  Data  Transfer  Objects  (DTOs)  like  Activity, TeamMember, and  JournalEntry. These  classes  handle  the

SQLite
 
mapping
 
arrays

Firebase
 
JSON

 ●  /lib/data/: Houses  local_repository.dart, the  core  SQLite  execution

tables
 
for
 
Trips,
 
Itinerary
 
Days,
 
and
 
Activities.
 ●  /lib/services/: Contains  the  core  business  logic  controllers.  ○  LocationSyncService.dart: Manages  live  telemetry,  handles  QR  
code
 
payload

for
 
joining
 
dynamic
 
groups,
 
and

real-time
 
pings
 
to
 
the
 
Firestore
 
database.
 ○  SyncService.dart: The  background  polling  agent  that  fetches  
Firestore
 
records
 
and
 
updates
 
the
 
SQLite
 
repository.
 ○  BadgeService.dart: Computes  geographic  proximity  using  
Haversine
 
formulas

user's
 
location

gamified
 
milestones.
 ○  JournalService.dart &  VaultService.dart: Manages  file  I/O

 ●  /lib/pages/: Contains  stateful  view  screens  governing  individual

 MapPage, ItineraryOverviewPage, GuideRepositoryPage). ●  /lib/widgets/: Highly  customized  components  meant  for  reuse,  such  as  
the
 TeamBottomSheet (handling  QR  generation/scanning)  and  TimelineCard. 
 
3.3.  Database  Schema  
Sojourn  Compass  relies  on  a  robust  data  dictionary  bridging  both  a  local  
relational
 
database
 
and
 
a
 
remote
 
NoSQL
 
structure.
 
Local  SQLite  Schema  (Offline  Cache)  
Table  Fields  
trips  tripId (TEXT,  Primary  Key),  tripName (TEXT),  startDate (TEXT),  endDate (TEXT),  destinations (TEXT),  status (TEXT)  
itinerary_days  dayId (TEXT,  Primary  Key),  tripId (TEXT,  Foreign  Key),  date (TEXT),  dayNumber (INTEGER),  theme (TEXT)  activities  activityId (TEXT,  Primary  Key),  dayId (TEXT,  Foreign  Key),  siteName (TEXT),  description (TEXT),  category (TEXT),  location (TEXT),  scheduledArrival (TEXT),  scheduledDeparture (TEXT),  whatToBring (TEXT  —  Serialized  JSON  array  for  the  Smart  Checklist)  
Cloud  Firestore  NoSQL  Schema  (Real-Time  Subsystems)  
Collection  Document  Structure  
users  Document  ID:  Firebase  Auth  UID  
teamId (String):  Reference  to  active  ad-hoc  group  
isTeamLeader (Boolean):  Grants  admin  privileges  for  group  tracking  
teams  Document  ID:  Auto-generated  string  
name (String):  Group  title  
leaderId (String):  Creator  UID  
joinCode (String):  6-digit  alphanumeric  key  for  UI  QR  generation  
Subcollection:  members  (Doc  ID  =  User  UID)  
- name (String),  role (String)  
- latitude (Double),  longitude (Double)  
- isOnline (Boolean)  
 
Subcollection:  pings  (Doc  ID  =  Ping  ID)  
- message (String)  
- timestamp (Timestamp)  journal_entries  id (String)  
title (String),  content (String)  
mediaUrl (String)  —  Cloudinary  Asset  link  
location (String)  
entryDate (Timestamp)  
vault_documents  id (String)  
userId (String)  —  Restricts  visibility  to  user  only  
title (String)  
fileUrl (String)  —  Firebase  Storage  URI  link  
category (String)  —  'Visas',  'Passports',  etc.  
activity_guides  id (String)  
title (String),  content (String)  
category (String),  location (String)  
mediaUrls (Array)  —  Links  to  audio  lectures
